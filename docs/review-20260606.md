## 三、分层边界审查

原报告把 `pkg/templatex` 依赖 `internal` 判断为“外部消费者无法编译”，该结论不准确。

当前可复核的依赖事实：

```go
// pkg/templatex/config.go
import "github.com/ZoneCNH/xlib-standard/internal/sanitize"
import "github.com/ZoneCNH/xlib-standard/internal/validation"
```

Go 的 `internal` 可见性规则限制的是导入语句所在包的位置。`pkg/templatex` 自身位于同一父级模块 `github.com/ZoneCNH/xlib-standard` 下，因此这些内部导入对该包实现是合法的。外部仓库导入 `github.com/ZoneCNH/xlib-standard/pkg/templatex` 时，并不会因为 `pkg/templatex` 的实现导入了本模块 `internal` 包而直接被 Go 规则禁止。

剩余风险不是编译阻断，而是治理/架构决策：如果本仓库规定公共 `pkg/` 包不得依赖 repo-local `internal/` helper，则应把该约束写入规则或 contract，并选择以下一种处理方式：

- 接受该实现依赖，并在公共包边界规则中明确允许。
- 将公共包需要复用的 helper 移到公共或包内可见位置。
- 为 `pkg/templatex` 增加外部消费者编译 smoke test，用 Evidence 证明边界行为。
