## 0. 结论先行

`xlib-standard` 的自动化依赖升级与 `kernel` 不同。  
`kernel` 是 L0 实现库，关注自身依赖与 API 契约；  
`xlib-standard` 是标准源，关注：

```text
标准文档 → 模板契约 → Generator → Harness Gate → Evidence Protocol → 下游同步
```

所以 `xlib-standard` 的自动化落地目标不是只升级 `go.mod` 或 GitHub Actions，而是让标准仓库变成：

> 所有基础库标准变更、依赖升级、模板漂移、下游同步的 SSOT 与触发器。

本方案推荐以 5 个 PR 落地：

| PR | 主题 | 目标 |
|---|---|---|
| PR-1 | Clean Naming / Role Boundary | 清理 `baselib-template` / `foundationx` 旧命名残留 |
| PR-2 | Add Dependency Bots | 接入 Renovate + Dependabot |
| PR-3 | Add Standard Diff / Downstream Impact Gate | 标准变更时自动识别影响 kernel / L1 / x.go |
| PR-4 | Strengthen Generator / Harness Evidence | generator、contracts、release manifest 都进入 Evidence |
| PR-5 | Add Release Train / Weekly Report | 自动生成依赖升级与标准漂移周报 |

## 1. 问题的底层本质

表层问题：

> xlib-standard 如何做自动化依赖升级？

底层问题：

> 如何让标准源仓库在自身依赖更新、标准文档变更、模板契约变更后，自动判断哪些下游必须同步、哪些 Gate 必须增强、哪些变更必须人工裁决。

`xlib-standard` 是整个基础库体系的标准源，影响链路是：

```text
xlib-standard
  ↓
kernel
  ↓
configx / observex / redisx / kafkax / postgresx / taosx / ossx / clickhousex
  ↓
x.go
```

因此，`xlib-standard` 的自动化必须回答 4 个问题：

1. 自己是否需要升级依赖？
2. 标准是否发生漂移？
3. 下游是否必须同步？
4. 是否可以自动合并？

## 2. 不可再拆解的基本真理

| 编号 | 基本真理 | 执行含义 |
|---|---|---|
| T-001 | xlib-standard 是标准权威源 | 标准变化必须可追踪 |
| T-002 | 模板实现不能替代标准源 | 标准正文与 generator 实现必须分离 |
| T-003 | 标准变更会影响下游 | 必须生成 downstream impact report |
| T-004 | Harness Gate 是 Evidence 来源 | Gate 变更必须记录原因 |
| T-005 | generator 变更风险高 | 必须跑代表性下游 smoke |
| T-006 | major / schema / contract 变更不能自动合并 | 需要 Human Approval |
| T-007 | 没有 Evidence 不能宣称 DONE | 所有标准升级必须有 release manifest |
| T-008 | xlib-standard 不能依赖业务 | 禁止引入 x.go 业务模型与生产凭据 |

## 3. 被误认为真理的常见假设

| 假设 | 问题 | 正确裁决 |
|---|---|---|
| xlib-standard 只是文档仓库 | 它承担标准、模板、Harness、Evidence SSOT | 必须有 Gate 与 release evidence |
| 标准文档改了不用测试 | 下游可能需要同步 | 必须生成 downstream impact |
| generator 只要本仓库测试通过 | 生成结果可能不可用 | 必须跑 representative downstream |
| Dependabot 足够 | 无法处理标准漂移和下游影响 | 需要 standard-impact-check |
| 所有 GitHub Actions 可自动合并 | major action 可能破坏 CI | major 需要 review |
| 标准源可包含真实 runtime | 会污染边界 | xlib-standard 不承载真实 L1 runtime |

## 4. 可以被打破的限制

| 旧限制 | 新方案 |
|---|---|
| 标准更新靠人工通知下游 | 标准变更自动生成 downstream impact report |
| 依赖升级只看 go.mod | 同时覆盖 GitHub Actions、工具链、Generator、Harness |
| 标准文档和实现不同步 | docs-check / generator-check / release-evidence-check |
| 下游同步口头约定 | `.standard-sync.yaml` + sync report |
| 大批量升级难审查 | packageRules + groupName + small batch |
| 完成声明缺证据 | DONE with evidence 固化 |

## 6. Spec

### SPEC-xlib-standard-dependency-upgrade-v1.0

#### REQ-001：标准源命名与边界清理

`xlib-standard` 不得继续混用旧名称导致标准源与模板实现边界不清。

验收标准：

| AC | 验收方式 |
|---|---|
| AC-001-001 | README 标题与仓库定位使用 `xlib-standard` |
| AC-001-002 | 文档中明确区分 Standard / Template / Generator / L0 / L1 / x.go |
| AC-001-003 | 旧名 `baselib-template` 只允许出现在迁移说明 |
| AC-001-004 | `foundationx` 如已裁决为 `kernel`，必须同步角色表 |
| AC-001-005 | docs-check 能发现旧命名残留 |

#### REQ-002：自动依赖发现

仓库必须能自动发现 Go module 和 GitHub Actions 依赖升级。

验收标准：

| AC | 验收方式 |
|---|---|
| AC-002-001 | 根目录存在 `renovate.json` |
| AC-002-002 | 存在 `.github/dependabot.yml` |
| AC-002-003 | 启用 `gomod` 与 `github-actions` |
| AC-002-004 | major 更新不自动合并 |
| AC-002-005 | generator / contract / standard 相关变更需要 review |

#### REQ-003：标准变更影响分析

每次 `docs/standard/`、`contracts/`、`Makefile`、`scripts/`、`.agent/` 变化，必须生成影响报告。

验收标准：

| AC | 验收方式 |
|---|---|
| AC-003-001 | 存在 `scripts/check_standard_impact.sh` |
| AC-003-002 | 生成 `release/standard-impact/latest.md` |
| AC-003-003 | 报告列出影响 downstream |
| AC-003-004 | 报告区分 docs / contracts / harness / generator / evidence |
| AC-003-005 | contract/harness/evidence 变化必须标记 `downstream-sync-required` |

#### REQ-004：Generator / Harness Gate 强化

Generator 变化必须证明模板能生成代表性下游。

验收标准：

| AC | 验收方式 |
|---|---|
| AC-004-001 | `make integration` 能跑 generator smoke |
| AC-004-002 | 必须渲染代表性下游：`kernel` 或 `corekit` |
| AC-004-003 | 生成结果无模板 token 残留 |
| AC-004-004 | 生成结果 `GOWORK=off go test ./...` 通过 |
| AC-004-005 | generator evidence 写入 manifest |

#### REQ-005：Evidence Manifest 扩展

release manifest 必须记录标准影响、依赖清单、工具版本、contract fingerprint、downstream impact。

验收标准：

| AC | 验收方式 |
|---|---|
| AC-005-001 | manifest 包含 `dependencies` |
| AC-005-002 | manifest 包含 `standard_impact` |
| AC-005-003 | manifest 包含 `downstream_sync_required` |
| AC-005-004 | manifest 包含 `tools` |
| AC-005-005 | `make release-evidence-check` 校验所有 evidence |

#### REQ-006：下游同步机制

标准变化后必须明确是否触发 `kernel` 同步。

验收标准：

| AC | 验收方式 |
|---|---|
| AC-006-001 | 存在 `docs/downstream-sync-policy.md` |
| AC-006-002 | 标准变化类型映射到下游同步动作 |
| AC-006-003 | `kernel` 同步规则单独列出 |
| AC-006-004 | L1 基础库同步规则单独列出 |
| AC-006-005 | x.go 仅作为消费方，不反向影响标准源 |

### 7.1 自动升级架构

```text
Dependency Sources
  ├─ Go modules
  ├─ GitHub Actions
  ├─ Go toolchain
  ├─ generator scripts
  ├─ contracts
  └─ standard docs
       ↓
Upgrade / Drift Detection
  ├─ Renovate
  ├─ Dependabot security alerts
  ├─ check_dependency_diff.sh
  └─ check_standard_impact.sh
       ↓
PR Classification
  ├─ patch
  ├─ minor
  ├─ major
  ├─ security
  ├─ contract-change
  ├─ harness-change
  ├─ generator-change
  └─ standard-change
       ↓
Harness Gates
  ├─ make ci
  ├─ make integration
  ├─ make docs-check
  ├─ make contracts
  ├─ make release-check
  └─ make release-final-check
       ↓
Evidence
  ├─ release/manifest/latest.json
  ├─ release/dependency/modules.txt

       ↓
Decision
  ├─ patch automerge
  ├─ major block
  ├─ contract review
  ├─ downstream sync issue
  └─ release
```

### 8.3 新增 scripts/check_dependency_diff.sh

```bash
#!/usr/bin/env bash
set -euo pipefail

ROOT="$(cd "$(dirname "${BASH_SOURCE[0]}")/.." && pwd)"
cd "$ROOT"

mkdir -p release/dependency

echo "== module list =="
GOWORK=off go list -m all | tee release/dependency/modules.txt

echo "== module updates =="
GOWORK=off go list -m -u all | tee release/dependency/updates.txt

GOWORK=off go mod tidy

if ! git diff --exit-code -- go.mod go.sum; then
  echo "ERROR: go.mod/go.sum changed after go mod tidy"
  exit 1
fi

echo "dependency diff check passed"
```

### 8.4 新增 scripts/check_standard_impact.sh

```bash
#!/usr/bin/env bash
set -euo pipefail

ROOT="$(cd "$(dirname "${BASH_SOURCE[0]}")/.." && pwd)"
cd "$ROOT"

mkdir -p release/standard-impact

REPORT="release/standard-impact/latest.md"

{
  echo "# xlib-standard standard impact report"
  echo
  echo "- generated_at: $(date -u '+%Y-%m-%dT%H:%M:%SZ')"
  echo "- repo: ZoneCNH/xlib-standard"
  echo
  echo "## Impact areas"
} > "$REPORT"

impact_required=0

check_path() {
  local label="$1"
  local pattern="$2"
  if git diff --name-only HEAD~1..HEAD 2>/dev/null | grep -E "$pattern" >/dev/null 2>&1; then
    echo "- ${label}: changed" >> "$REPORT"
    impact_required=1
  else
    echo "- ${label}: unchanged" >> "$REPORT"
  fi
}

check_path "standard_docs" '^docs/standard/'
check_path "contracts" '^contracts/'
check_path "harness" '^(Makefile|scripts/check_|\.github/workflows/)'
check_path "generator" '^scripts/render_template\.sh'
check_path "agent_runtime" '^\.agent/'
check_path "release_evidence" '^(scripts/generate_manifest\.sh|scripts/check_release_evidence\.sh|docs/release\.md)'

{
  echo
  echo "## Downstream sync"
  if [ "$impact_required" -eq 1 ]; then
    echo "- downstream_sync_required: true"
    echo "- primary_downstream: github.com/ZoneCNH/kernel"
    echo "- labels: standard-change, downstream-sync-required"
  else
    echo "- downstream_sync_required: false"
  fi
} >> "$REPORT"

cat "$REPORT"
echo "standard impact check passed"
```

## 13. Rollback Protocol

如果依赖升级或标准变更合并后造成下游失败：

```text
1. 立即冻结相关 packageRule 或标准变更路径
2. revert xlib-standard 变更 PR
3. 创建 downstream incident issue
4. 标记 affected repos
5. 生成 standard-impact postmortem
6. 新增 Harness Patch 或 Rule Patch
```
